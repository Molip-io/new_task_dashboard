# 업무현황·의사결정 대시보드 시스템 검증 요청서

문서 버전: 3.0  
기준일: 2026-07-14  
검증 대상: 업무현황 수집, 프로젝트 진행 판정, 병목·의사결정 탐지, GPT 요약, 대시보드 표시 전 과정

## 검증 후 반영 결정

1차 검증의 `조건부 가능` 판정을 수용하고 아래를 구현 전 필수 조건으로 반영했다.

- Slack은 2단계 선택 기능이 아니라 v1 필수 출처다. Notion 단일 소스 파일럿으로 축소하지 않는다.
- Slack 메시지·스레드·permalink와 원문 발췌를 보존하고, 대표가 대시보드에서 실제 내용을 확인할 수 있게 한다.
- Slack 자연어 해석 결과는 원문과 분리된 `주장 후보`로 저장한다. 모호한 문장은 상태·병목 수치를 바꾸지 않는다.
- 핵심 스펙마다 범위 확정 예정일·제작 완료 예정일·최종 목표일을 두어 계획 대비 진행 기준선을 만든다.
- 의존관계 템플릿은 확정 관계가 아니라 후보를 생성한다. 검토율을 첫 화면에 표시한다.
- 결정을 일급 엔티티로 추가하고 `대기 → 결정됨 → 전파됨 → Notion 반영 확인 → 종료` 수명주기를 관리한다.
- 확인 필요 항목은 핵심 필드 충돌에만 생성한다. 기한이 지나도 자동 삭제하지 않고 `기한 초과`로 재판정한다.
- 절대적인 `7일 무편집` 경고는 v1에서 제거한다.

## 1. 검증 요청 요약

이 시스템은 소규모 게임 스튜디오 대표가 여러 프로젝트의 진행 상황과 의사결정 우선순위를 짧은 시간 안에 파악하도록 돕는 업무 대시보드다.

v1의 기본 사용자는 대표다. 담당별 흐름 탭은 팀원 평가 화면이나 팀원용 셀프서비스가 아니라, 대표가 자식 일감 기준 전원의 현재 업무량과 사람 사이의 대기 관계를 함께 보는 화면이다.

검증자는 아래 질문에 답해주기 바란다.

1. 프로젝트가 현재 어디까지 진행됐고 계획 대비 어느 정도인지 판단할 수 있는가?
2. 프로젝트 건강 상태를 단순 완료율이 아니라 핵심 스펙·의존관계·후속 영향으로 판단하는가?
3. 대표가 오늘 결정할 것과 결정 전에 확인할 정보를 구분할 수 있는가?
4. 누가 누구를 기다리는지와 팀 전체 대기 시간이 드러나는가?
5. Notion·Slack·회의록이 충돌할 때 잘못된 단일 결론을 만들지 않는가?
6. GPT가 사실을 바꾸지 않고 읽기 좋은 브리핑만 생성하도록 통제되는가?

칭찬보다 오판 가능성, 누락, 운영 실패 지점을 중심으로 검토해주기 바란다.

## 2. 배경과 문제

- 프로젝트: 피자레디, 포지 앤 포춘, Color Fill 등 여러 게임 프로젝트가 동시에 진행된다.
- 파트: 기획, 개발, 아트, QA, 운영 등이 협업한다.
- 데이터는 Notion 작업 DB·프로젝트 DB·회의록과 Slack 프로젝트 채널에 나뉘어 있다.
- 노션에는 공식 작업과 스펙이 있지만 실제 논의·변경은 Slack이나 회의에서 먼저 발생할 수 있다.
- 대표는 작업 목록을 많이 보는 것이 아니라 프로젝트 진행 가능성과 개입 시점을 빠르게 판단해야 한다.

현재의 핵심 문제는 `작업이 몇 건 진행 중인가`가 아니라 다음과 같다.

- 프로젝트가 목표 대비 어느 단계에 있는지 한눈에 파악하기 어렵다.
- 핵심 스펙이 확정됐는지와 후속 파트에 미치는 영향이 분리되어 있지 않다.
- 개발이 기획 확정 전에 진행되는 재작업 위험과 실제 대기가 혼동된다.
- 결정이 늦어져 팀이 기다리는 상황을 개인 작업량 화면에서는 알기 어렵다.
- 서로 다른 출처의 충돌을 최신 시각만으로 합치면 잘못된 현황이 될 수 있다.

## 3. 시스템 목표

### 3.1 핵심 목표

대표가 각 프로젝트에 대해 다음을 5분 안에 판단하게 한다.

1. 현재 어느 단계인가?
2. 계획 대비 얼마나 진행됐는가?
3. 목표일을 지킬 가능성은 어떤가?
4. 무엇이 실제로 막혀 있는가?
5. 오늘 무엇을 결정하거나 확인해야 하는가?
6. 누구의 결정이나 작업 때문에 누가 기다리는가?
7. 어제와 비교해 무엇이 달라졌는가?

### 3.2 성공 기준

- 첫 화면 30초 스캔으로 오늘 결정할 안건과 실제 병목을 식별할 수 있다.
- 프로젝트 탭에서 현재 단계, 목표 마일스톤, 핵심 스펙 상태, 파트별 진행을 확인할 수 있다.
- 모든 결정·병목·충돌 문장에 원본 근거 링크가 있다.
- GPT가 실패해도 프로젝트 상태, 병목, 결정 큐, 전일 변화는 표시된다.
- 출처 충돌을 자동으로 덮어쓰지 않고 확인 담당자와 기한을 만든다.
- 개인 화면은 요약 대상의 모든 담당자를 빠짐없이 보여주며, 자식 일감 수·기한 초과·기한 없음·스펙 쏠림과 대기 관계를 함께 표시한다. 이 수치는 성과 평가에 사용하지 않는다.
- Slack에서 발견된 결정·범위·기한·담당·대기·블로커·상태 변화의 원문과 해석 상태를 확인할 수 있다.
- Slack 수집 또는 의존관계 검토가 부족하면 `병목 없음` 대신 판정 커버리지 부족을 표시한다.

### 3.3 비목표

- 팀원 성과 평가 또는 개인 순위 산출
- Slack 메시지 수를 생산성 지표로 사용
- 단순 작업 완료율로 프로젝트 건강 상태 판정
- GPT가 프로젝트 상태나 사실관계를 임의로 결정
- 실시간 감시 시스템 구축. 기본 갱신은 하루 1회이며 필요 시 수동 갱신한다.
- 최상위 스펙의 공동 담당자를 개인별 상세 일감 수로 중복 계산하는 것

## 4. 프로젝트 진행 정도의 정의

진행 정도와 프로젝트 건강 상태는 별도 값이다. 완료율이 높아도 핵심 스펙이 미확정이면 위험할 수 있고, 초기 단계라도 계획대로라면 정상일 수 있다.

| 관점 | 표시 요소 | 검증 목적 |
|---|---|---|
| 현재 단계 | 기획 중, 범위 확정, 제작 중, 검토, QA, 출시 준비 | 프로젝트의 현재 위치 파악 |
| 목표 마일스톤 | 범위 확정 예정일, 제작 완료 예정일, 최종 목표일, D-day, 계획 대비 앞섬·정상·지연 | 일정 달성 가능성 판단 |
| 핵심 스펙 | 확정 수/전체, 완료 수/전체, 미확정 목록 | 목표 범위 안정성 판단 |
| 파트별 진행 | 기획·개발·아트·QA별 완료·진행·대기 | 파트 흐름 파악 |
| 크리티컬 패스 | 완료 단계/전체 단계, 현재 막힌 관계, 잔여 핵심 작업 | 출시까지 실제 남은 흐름 파악 |
| 전일 변화 | 단계·기한·담당·의존관계·결정 변화 | 정체와 진행 속도 감지 |
| 판정 커버리지 | 핵심 스펙 의존관계 검토율, Slack 채널 수집 성공률, 출처별 마지막 성공 시각 | 문제 없음과 판단할 데이터 없음 구분 |

작업 완료율은 참고용으로만 표시하며 프로젝트 건강 상태 판정에는 직접 사용하지 않는다.

## 5. 데이터 수집 및 운영 워크플로우

```text
GitHub Actions · 매일 07:30 KST
        ↓
Notion 작업·스펙·회의 본문 + Slack 메시지·스레드·링크 수집
        ↓
Slack 원문에서 결정·범위·기한·담당·의존관계·블로커·상태 변화 후보 추출
  - 원문 발췌·작성자·시각·채널·thread permalink 보존
        ↓
프로젝트 → 핵심 스펙 → 파트 작업 → 의존관계 후보·확정 관계 정규화
        ↓
출처별 주장 보존 및 충돌 탐지
        ↓
규칙 엔진이 단계·상태·실제 대기·재작업 위험·영향·전일 변화 계산
        ↓
GPT가 계산된 사실을 대표 브리핑 문장으로 요약
        ↓
스키마·ID·근거 검증 후 dashboard 데이터 게시
        ↓
대표가 오전 대시보드 확인
```

### 운영 원칙

- 자동 실행 담당은 GitHub Actions 예약 워커다.
- 컴퓨터가 꺼져 있어도 수집되어야 한다.
- 수동 새로고침도 동일 파이프라인을 한 번 실행한다.
- 수집 실패 시 전날 성공 데이터와 소스별 실패 경고를 함께 보여준다.
- 소스별 마지막 성공 시각과 전체 기준 시각을 표시한다.
- GPT 실패는 데이터 수집과 규칙 판정을 실패시키지 않는다.
- Slack 추출 실패는 Slack 원문 수집을 폐기하지 않는다. 원문은 표시하고 주장 후보만 `생성 안 됨`으로 표시한다.
- Slack 일부 채널 실패 시 실패 채널과 마지막 성공 시각을 프로젝트별로 표시한다.

## 6. 기준 데이터 모델

### 6.1 프로젝트

- 프로젝트 ID, 이름, 범위 확정 예정일, 제작 완료 예정일, 최종 목표일, 현재 단계, 실제 단계 전환 시각
- 핵심 스펙 목록, 프로젝트 책임자, Slack 채널, 회의록 연결
- 건강 상태와 확인 상태를 별도로 관리

### 6.2 스펙

- 핵심/지원 스펙 구분
- 범위 확정 여부와 작업 진행 상태 분리
- 범위 확정 예정일, 제작 완료 예정일, 최종 목표일, 완료 조건, 파트 작업 목록

### 6.3 파트 작업

- 프로젝트·스펙·파트·담당자·상태·기간
- 선행 작업, 필요한 선행 상태, 대기 사유
- 결정 안건과의 연결

### 6.4 출처별 주장

- 출처: Notion, Slack, 회의록
- 대상: 프로젝트, 스펙, 작업, 결정
- 필드: 상태, 기한, 담당, 범위, 의존관계, 결정, 블로커
- 원문 발췌, 작성자, 발생 시각, 수집 시각, 원본 링크

### 6.5 의존관계

- 관계 대상, 필요한 선행 상태, 후속 작업, 영향 파트
- 상태: `후보`, `확정 관계`, `관계 없음 확인`, `미검토`
- 프로젝트 템플릿은 후보만 자동 생성하고 사람이 확정하거나 관계 없음을 확인한다.
- 커버리지는 핵심 스펙별 후보 검토 완료 여부로 계산한다.

### 6.6 결정

- 질문, 프로젝트·스펙, 결정자, 기한, 선택지, 입력에 존재하는 권고안, 영향 작업
- 상태: `대기`, `결정됨`, `전파됨`, `Notion 반영 확인`, `종료`, `재개방`
- Slack에서 지정 결정자가 명시적으로 결정한 경우 원문 근거를 연결해 `결정됨·Notion 미반영`으로 기록한다.
- Notion·Slack·회의록에서 이미 발생한 결정을 연결하는 내부 조회 모델이며, 대표 브리핑에서 결정을 입력하거나 저장하지 않는다.

### 6.7 Slack 근거 신호

- 메시지 ID, `thread_ts`, 채널, 작성자, 작성 시각, permalink, 원문 발췌
- 연결 프로젝트·스펙·작업과 연결 방식
- 주장 유형: 결정, 범위, 기한, 담당, 의존관계, 블로커·대기, 상태 변화
- 명시성: `명시적 원문`, `해석 후보`
- 확인 상태: `미확인`, `사람 확인`, `Notion 반영 확인`, `서로 다른 맥락`

## 7. 상태·병목 판정 원칙

### 7.1 프로젝트 상태

`개입필요` 조건:

- 핵심 스펙이 미결 결정 또는 확인된 선행 의존성 때문에 막혀 있다.
- 오늘까지 결정해야 하는 안건이 실제 팀 대기를 발생시킨다.
- 핵심 마일스톤이 초과됐다.
- 하나의 블로커가 2명 이상 또는 2개 파트 이상에 영향을 준다.

`주의` 조건:

- 확인된 실제 대기 관계가 있다.
- 핵심 스펙 시작이 3일 이내인데 범위가 미확정이다.
- 선행 조건 충족 전 후속 작업이 진행되어 재작업 위험이 있다.
- 핵심 필드의 출처 충돌이 확인 기한을 초과했다.

### 7.2 의존관계 판정

- 선행 미충족 + 후속 작업 `대기` → 실제 대기
- 선행 미충족 + 후속 예정일 초과 및 미착수 → 추정 대기
- 선행 미충족 + 후속 작업 진행·검토·완료 → 순서 위반 및 재작업 위험
- 명시 관계 없음 + 파트 순서만 추정 → 의존관계 확인 필요, 병목 수치 제외
- Slack에 명시적 대기 문장이 있지만 관계 미등록 → `보고된 대기`, 대표 확인 우선순위에는 포함하되 실제 대기 인원·인일 계산에서는 제외

의존관계 후보와 미검토 관계는 `핵심 스펙 의존관계 검토율`을 낮춘다. 커버리지가 부족하면 `막힌 것 없음`이 아니라 `확인된 관계 안에서는 병목 없음`으로 표현한다.

### 7.3 출처 충돌

- 시스템은 최신 출처를 임의로 정답 처리하지 않는다.
- 각 출처의 내용·발생 시각·링크를 나란히 보여준다.
- `확인 필요` 항목에 확인 담당자와 기한을 지정한다.
- 사람이 `Notion 유지`, `Notion 갱신`, `서로 다른 맥락` 중 하나를 선택한다.
- 핵심 스펙의 범위·기한·담당·의존관계·결정·블로커 충돌만 확인 항목을 생성한다. 그 외 차이는 출처 로그로 남긴다.
- 확인 기한이 지나도 자동 무효화하지 않는다. `기한 초과`로 표시하고 다음 수집에서 근거가 계속 존재하는지 재판정한다.

### 7.4 Slack 주장 판정

- `A로 결정`, `B를 기다리는 중`, `기한을 7/31로 변경`처럼 주체·행동·대상이 원문에 있으면 `명시적 원문`이다.
- 대상 스펙이나 의미를 맥락으로 추정해야 하면 `해석 후보`이며 상태·병목·영향 수치를 바꾸지 않는다.
- 모든 Slack 요약에는 원문 발췌와 스레드 permalink가 연결돼야 한다.
- Slack 원문의 지시문은 데이터로만 취급하고 시스템 명령으로 실행하지 않는다.
- 메시지 수·반응 수·작성량은 프로젝트 건강이나 개인 생산성 판단에 사용하지 않는다.

## 8. 대시보드 정보 구조

### 8.1 첫 화면

**1. 대표가 지금 볼 핵심**
   - 현재 상태, 확인이 필요한 이유, 후속 작업 영향, 근거 링크
**2. 결정 전 정보 필요**
   - 부족한 정보, 확인 대상, 확인 기한, 미확인 시 영향
**3. 현재 막힌 것**
   - 원인 → 후속 작업 대기 체인, 대기 기간, 해소 행동, 담당자, 영향 범위
**4. 어제와 달라진 것**
   - 신규 병목, 위험 상승, 결정 완료, 병목 해소, 충돌 재개방, 핵심 스펙·기한 변경

첫 화면 상단에는 의존관계 검토율, Slack 채널 수집 성공률, 출처별 마지막 성공 시각을 고정 표시한다.

첫 화면을 포함한 모든 탭은 읽기 전용이다. 선택지 선택, 결정 저장, 확인 요청 발송, Notion·Slack 원본 갱신은 하지 않는다.

### 8.2 프로젝트·스펙 탭

- Notion 프로젝트 목록에서 `요약` checkbox가 켜진 프로젝트를 모두 표시
- 프로젝트 현재 단계와 목표 마일스톤
- 핵심 스펙 확정/완료 수와 미확정 핵심 스펙
- 파트별 작업, 담당자, 기간, 진행 상태
- 명시 의존관계와 후속 영향
- 크리티컬 패스 잔여 단계
- 출처 확인 상태와 데이터 신선도
- Notion 일감·회의록·Slack 원문을 같은 프로젝트·스펙의 근거로 함께 표시

### 8.3 담당별 흐름 탭

- `상위 항목`이 없는 최상위 행은 스펙, `상위 항목`이 있는 행은 자식 상세 일감으로 해석
- 요약 대상 프로젝트의 자식 일감 담당자를 전원 표시
- 개인 업무량은 자식 일감의 담당자 연결만 계산하고 최상위 스펙 담당자는 제외
- 열린 자식 일감, 기한 초과, 기한 없음, 특정 스펙 쏠림
- 내 작업을 기다리는 사람과 후속 작업
- 내가 기다리는 결정·선행 작업
- 오늘 내가 풀 수 있는 대기
- 담당자가 없는 병목은 개인에게 귀속하지 않고 팀 큐로 표시

개인의 작업 수는 병목 후보를 찾기 위해 표시하지만 난이도·생산성·성과 점수로 해석하지 않는다. 공동 담당 일감은 각 개인에게 1건씩 연결하므로 전체 자식 일감 수와 담당 연결 수를 구분한다.

### 8.4 프로젝트 안의 Slack 근거

- Slack은 별도 최상위 탭을 만들지 않고 관련 프로젝트·스펙과 확인 필요 항목에 연결한다.
- 대표 판단과 연결된 Slack 원문 발췌
- 작성자·시각·채널·스레드 permalink
- 연결된 프로젝트·스펙·작업
- 추출 주장과 `명시적 원문/해석 후보` 상태
- 사람 확인·Notion 반영 여부

단순 채널별 메시지 수만 표시하지 않으며, 원문 근거가 없는 Slack 요약은 화면에 올리지 않는다.

### 8.5 확인 필요 탭

- 서로 다른 출처별 주장
- 확인 담당자와 확인 기한
- 확인 결과와 Notion 반영 상태

## 9. GPT 요약 역할과 통제

Slack 추출 AI는 제한된 주장 후보와 원문 연결만 만들고, 브리핑 GPT는 프로젝트 상태·실제 대기·영향 인원·기한을 계산하지 않는다. 규칙 엔진이 계산한 사실과 프로젝트에 연결된 Slack 근거를 대표가 읽기 좋은 문장으로 압축한다.

### GPT가 작성하는 내용

- 전체 헤드라인
- 결정 안건의 `왜 지금`
- 입력에 존재하는 권고안의 요약
- 병목·전일 변화 설명
- 프로젝트 한 줄 요약
- 프로젝트에 연결된 Slack 근거의 짧은 설명

### GPT 금지 사항

- 입력에 없는 사람·수치·기한·관계 생성
- 출처 충돌의 임의 해결
- 명시되지 않은 의존관계를 실제 병목으로 표현
- 정보가 부족한 안건에 결정을 강요
- 개인 작업량을 성과·생산성 평가로 해석하거나 점수화
- Slack `해석 후보`를 확정 결정·실제 병목·확정 일정으로 승격

### 검증 방식

- 모든 서술은 입력 객체 ID를 참조한다.
- 모든 서술에는 존재하는 근거 ID가 연결돼야 한다.
- JSON Schema에 맞지 않으면 한 번 재시도 후 고정 템플릿을 사용한다.
- GPT 실패 시에도 규칙 facts는 그대로 표시한다.

실제 Slack 주장 추출 지침은 `prompts/slack-claim-extractor.md`, 출력 계약은 `schemas/slack-claim-extraction.schema.json`에서 관리한다. 브리핑 GPT 행동 지침은 `prompts/dashboard-summary.md`, 출력 계약은 `schemas/dashboard-summary.schema.json`에서 별도 관리한다.

## 10. 대표 사례: 피자레디 익스프레스

입력 상황:

- 스펙: 익스프레스 기능, 핵심 스펙, 목표 7월 31일
- 기획: 익스프레스 기획 중, 담당 a, 7월 11일~31일, 핵심 범위 미확정
- 개발: 기획 완료 전 일부 선행 개발 중, 담당 b, 기한 미등록
- 아트: 머신 작업 중, 담당 c, 의존관계 미등록
- Notion: 기획 중, 7월 31일 완료 예정
- Slack: a가 `개발이 사용하는 핵심 플로우는 이 안으로 잠그자`고 작성, 11:20, 스레드 permalink 존재
- 회의록: 전체 범위 축소 논의

기대 판정:

- 프로젝트 진행: 핵심 스펙 범위 확정 단계, 일부 개발 선행 진행
- 개발 b: 실제 대기가 아니라 `선행확정 전 진행·재작업 위험`
- 아트 c: 의존관계 미등록이므로 병목 수치에서 제외하고 확인 필요
- 출처 충돌: 어느 출처도 자동 우선하지 않고 확인 담당 a, 확인 기한 설정
- Slack 판정: 지정 결정자의 명시적 문장이므로 `결정됨·Notion 미반영` 후보로 표시하고 원문을 화면에서 직접 확인
- 대표 결정: 전체 기획 완료 여부가 아니라 개발에 필요한 최소 변경 불가 범위 확정

기대 화면 문장 예시:

> 익스프레스는 핵심 범위 확정 단계이며 목표일은 7월 31일입니다. 개발은 일부 선행 진행 중이라 실제 대기는 아니지만 범위 변경 시 재작업 위험이 있습니다. 오늘 최소 범위를 확정하고 아트 의존관계를 확인해야 합니다.

Slack 원문 카드 예시:

> #몰입_피자레디 · a · 11:20  
> “개발이 사용하는 핵심 플로우는 이 안으로 잠그자. 나머지 연출은 기획하면서 조정.”  
> 추출 주장: 최소 개발 범위 결정 · 명시적 원문 · Notion 반영 대기

4. 설계를 더 단순하게 만들 수 있는 대안

## 14. 현재 구현 상태와 알려진 간극

현재 코드는 샘플 대시보드와 기본 수집 골격 수준이다. 다음 항목은 설계됐지만 아직 실제 연결되지 않았다.

- GitHub Actions 일일 실행 워크플로우
- Notion relation·rollup·회의 본문 수집
- Slack 스레드·permalink 수집
- Slack 원문 제한 추출, 명시성 등급, 프로젝트·스펙 연결, 사람 확인 상태
- 프로젝트·스펙·파트·의존관계 정규화
- 범위 확정·제작 완료·최종 목표 기준선과 계획 단계 비교
- 의존관계 후보 템플릿과 핵심 스펙 검토율
- 규칙·충돌·전일 변화 엔진
- GPT 요약 실행 및 JSON Schema 검증
- 결정 엔티티와 기록·전달·Notion 반영 확인 수명주기
- 새 의사결정 중심 UI와 실제 API 연결

따라서 이번 검증은 완성된 제품의 사용성 테스트가 아니라, 구현 전에 시스템 목표·판정 원칙·정보 구조·운영 흐름의 결함을 찾기 위한 설계 검증이다.
