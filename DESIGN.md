## 기능 적합성 검토 반영 결정 · 2026-07-15

### 수집 범위 예외

- 상태가 `일시 정지`인 스펙과 일감은 대시보드 수집 결과에서 제외한다.
- 일시 정지된 상위 스펙의 하위 일감도 상태와 무관하게 함께 제외해 프로젝트·완료율·담당별·AI 요약·스냅샷에 노출하지 않는다.

제품 관점 검토에서 `부분 적합` 판정을 받았으며, 아래 항목을 v1의 P0로 확정한다.

1. **계획 대비 기준선**: 스펙별 `범위 확정 예정일`과 `제작 완료 예정일`이 없으면 정상·지연을 판정하지 않는다.
2. **의존관계 커버리지**: 확정 관계 또는 `관계 없음 확인`이 없는 핵심 스펙은 병목 없음이 아니라 `판단 정보 부족`으로 표시한다.
3. **데이터 신선도**: Notion·Slack·회의록별 마지막 성공 시각과 부분 실패를 첫 화면에 고정 표시한다.
4. **전일 스냅샷**: 상태·기한·담당·완료율의 동일 필드를 날짜별로 저장하고, 이전 스냅샷이 없으면 변화 영역을 비운다.

기존 기능 중 건강상태 판정표, 결정 안건 승격 규칙, 확인 필요 화이트리스트, GPT 근거·대체 출력은 별도 메뉴가 아니라 해당 기능의 필수 정의로 포함한다. 병목 영향도는 막힘 카드에, 미배정 큐는 담당별 흐름에, 프로젝트 목표는 프로젝트 카드에 통합한다.

## 시스템 목표

이 시스템의 1차 목적은 대표가 각 프로젝트가 `어디까지 진행됐고`, `계획 대비 어느 정도 진행됐으며`, `다음 진행을 위해 무엇을 결정하거나 해소해야 하는지` 빠르게 파악하게 하는 것이다.

1. **현재 위치 파악**: 프로젝트와 핵심 스펙이 기획·확정·제작·검토·QA·출시 중 어느 단계인지 보여준다.
2. **계획 대비 진행 파악**: 목표 마일스톤, D-day, 일정 편차, 핵심 스펙 확정/완료 수, 크리티컬 패스 잔여 작업을 보여준다.
3. **목표 달성 가능성 판단**: 완료율만 보지 않고 미확정 핵심 스펙, 의존관계, 후속 영향으로 일정 위험을 판단한다.
4. **오늘의 의사결정 식별**: 지금 결정 가능한 안건과 결정 전에 정보가 필요한 안건을 구분한다.
5. **팀 대기 최소화**: 누가 누구의 결정·작업을 기다리는지와 오늘 해소할 행동을 보여준다.
6. **정보 신뢰성 확보**: Notion·Slack·회의록이 다르면 합치지 않고 출처별 주장과 확인 담당자를 표시한다. Slack 수집이 실패하거나 원문 연결이 부족하면 `병목 없음`이 아니라 `판단 범위 부족`으로 표시한다.
7. **변화 감지**: 어제와 비교해 새로 생긴 병목, 위험 상승, 결정 완료, 병목 해소, 기한 변경을 보여준다.

### 진행 정도의 표현

진행 정도와 프로젝트 건강 상태는 별도 값으로 관리한다. `80% 완료`여도 핵심 스펙이 미확정이면 위험할 수 있고, `30% 완료`여도 계획대로라면 정상일 수 있다.

| 진행 정보 | 화면 표현 | 용도 |
|---|---|---|
| 현재 단계 | 기획 중 · 범위 확정 · 제작 중 · 검토 · QA · 출시 준비 | 프로젝트가 어디에 있는지 파악 |
| 목표 마일스톤 | 범위 확정 예정일, 제작 완료 예정일, 최종 목표일, D-day, 계획 대비 앞섬/정상/지연 | 일정 가능성 판단 |
| 핵심 스펙 | 확정 수/전체, 완료 수/전체, 미확정 핵심 목록 | 목표 범위 안정성 판단 |
| 파트 진행 | 기획·개발·아트·QA별 완료/진행/대기 | 파트별 흐름 확인 |
| 크리티컬 패스 | 완료 단계/전체 단계, 현재 막힌 간선, 잔여 작업 | 출시까지 실제 남은 흐름 확인 |
| 최근 변화 | 전일 대비 단계·기한·담당·의존관계 변화 | 진행 속도와 정체 감지 |
| 판정 커버리지 | 핵심 스펙 의존관계 검토율, Slack 채널 수집 성공률, 출처별 마지막 성공 시각 | `문제 없음`과 `판단할 데이터 없음` 구분 |

단순 작업 완료율은 보조 지표로만 표시하고 프로젝트 상태 판정에는 직접 사용하지 않는다.

## v1 출처 정책과 Slack 처리

Slack은 보조 링크가 아니라 실제 진행 상황을 완성하는 필수 출처다. 다만 자연어를 해석한 결과가 공식 상태를 몰래 바꾸지 않도록 원문과 추출 주장을 분리한다.

### Slack에서 수집·표시하는 내용

- 프로젝트에 등록된 채널의 메시지와 스레드 답글
- 작성자, 작성 시각, 채널, `thread_ts`, permalink, 원문 발췌
- 프로젝트·스펙·작업 연결 결과와 연결 방식(`명시 ID`, `키워드`, `미연결`)
- 아래 7개 유형에 해당하는 주장 후보: `결정`, `범위`, `기한`, `담당`, `의존관계`, `블로커·대기`, `상태 변화`
- 추출 상태: `명시적 원문`, `해석 후보`, `사람 확인`, `Notion 반영 확인`

메시지 수·반응 수·작성량은 생산성이나 프로젝트 건강 상태 계산에 사용하지 않는다.

### Slack 주장 등급

| 등급 | 조건 | 시스템 처리 |
|---|---|---|
| 명시적 원문 | `A로 결정`, `B를 기다리는 중`, `기한을 7/31로 변경`처럼 주체·행동·대상이 원문에 존재 | 원문 카드와 운영 신호에 표시. 지정 결정자의 명시 결정은 `결정됨·Notion 미반영`으로 전환 가능 |
| 해석 후보 | 맥락을 읽어야 의미가 정해지거나 대상 스펙이 불명확 | `해석 확인 필요`로 표시하고 상태·병목·영향 수치 계산에서 제외 |
| 사람 확인 | 스펙 책임자나 결정자가 원문 해석과 연결 대상을 확인 | 확인된 주장으로 규칙 엔진 입력 가능 |
| Notion 반영 확인 | Slack 결정·변경이 공식 Notion 필드에 반영됨 | 결정 수명주기를 닫고 충돌·반영 대기 항목 종료 |

Slack의 명시적 대기 보고는 `보고된 대기`로 별도 표시한다. Notion의 확정 의존관계나 사람 확인이 연결되기 전까지 `확인된 실제 대기` 인원·인일에는 합산하지 않지만, 대표의 확인 우선순위에는 포함한다.

### 부분 실패 처리

- Slack 전체 실패: 전일 성공 데이터는 `오래된 Slack`으로 유지하고 상단에 `Slack 미수집 · 판정 범위 제한` 경고를 표시한다.
- 일부 채널 실패: 실패 채널과 마지막 성공 시각을 표시하고 해당 프로젝트는 `완전 수집`으로 표시하지 않는다.
- permalink 실패: 원문 ID와 채널·시각은 유지하되 `링크 확인 필요`로 표시한다.
- 추출 AI 실패: 원문 수집은 유지하고 Slack 주장·요약만 `생성 안 됨`으로 표시한다. Notion 규칙 판정은 계속 실행한다.
- Slack 원문 안의 명령문은 데이터로만 취급하며 시스템 지시로 실행하지 않는다.

## 핵심 데이터 모델 보완

### 프로젝트·스펙 진행 기준선

- `scopeFreezePlannedAt`: 범위 확정 예정일
- `productionCompletePlannedAt`: 제작 완료 예정일
- `targetAt`: 최종 목표일
- `currentStage`, `stageChangedAt`: 현재 실제 단계와 전환 시각
- 오늘의 계획 단계와 실제 단계를 비교해 앞섬·정상·지연을 판정한다. 단순 목표일 카운트다운만으로 정상 여부를 판단하지 않는다.

### 의존관계와 커버리지

- 프로젝트 유형별 템플릿은 의존관계 `후보`를 자동 생성한다.
- 사람은 후보를 `확정 관계` 또는 `관계 없음 확인`으로 바꾼다.
- 후보와 미검토 관계는 병목 수치에서 제외하되 `핵심 스펙 의존관계 검토율`에 반영한다.
- 첫 화면에는 핵심 스펙 기준 의존관계 검토율과 미검토 스펙 수를 항상 표시한다.

### 결정 엔티티

- 질문, 프로젝트·스펙, 결정자, 기한, 선택지, 입력에 존재하는 권고안, 영향 작업 링크
- 상태: `대기 → 결정됨 → 전파됨 → Notion 반영 확인 → 종료` 또는 `재개방`
- Slack에서 지정 결정자가 명시적으로 선택한 경우 원문 근거를 연결해 `결정됨·Notion 미반영`으로 기록한다.
- 결정 후 영향 작업이나 Notion 필드가 갱신되지 않으면 `반영 대기`로 첫 화면에 유지한다.
- 이 엔티티는 Notion·Slack·회의록에서 이미 발생한 결정을 읽고 연결하기 위한 내부 조회 모델이다.
- 대표 브리핑은 읽기 전용이다. 화면에서 선택하거나 결정을 저장하고, 확인 요청을 발송하거나 원본을 갱신하는 기능은 제공하지 않는다.

### 결정 안건 승격 규칙

`확인 필요`는 데이터 무결성 큐이고 `오늘 결정할 것`은 사업 판단 큐다. 같은 원본을 참조할 수 있지만 목적과 표시 내용을 구분한다.

- 핵심 스펙의 목표일·범위·출시 조건처럼 대표의 선택이 후속 작업을 바꾸는 항목만 `오늘 결정할 것`로 승격한다.
- 담당자 이름 매핑, 수집 실패, 관계 미입력처럼 사실 확인으로 닫히는 항목은 `확인 필요`에만 둔다.
- 출처 충돌 중 대표의 선택이 필요한 경우 브리핑에는 질문·기한·영향만 요약하고, 출처별 원문 비교는 `확인 필요`에서 보여준다.
- 결정 안건은 영향받는 스펙 또는 후속 작업, 기한, 근거가 모두 있을 때만 승격한다.
- 첫 화면에는 최대 5건만 표시하고 나머지는 `외 N건`으로 접는다.

### 전일 스냅샷과 변화

- 매일 오전 수집 성공 후 프로젝트·스펙의 상태, 기한, 담당, 완료율, 의존관계, 결정 상태를 날짜별 스냅샷으로 저장한다.
- 변화는 동일 ID와 동일 필드의 이전 값과 현재 값을 비교해 생성한다.
- 전일 스냅샷이 없거나 일부 출처가 실패한 경우 해당 범위의 변화를 만들지 않고 이유를 표시한다.
- Slack의 변화 언급만으로 `어제와 달라진 것`을 만들지 않는다. 원문은 프로젝트 근거 또는 확인 필요 후보로 유지한다.

## 상태 판정 규칙

프로젝트 상태는 작업 완료율이 아니라 핵심 스펙 확정 여부, 명시된 의존관계, 후속 작업 영향으로 계산한다. AI는 상태를 쓰거나 변경하지 않고, 규칙 결과를 문장으로만 설명한다.

| 조건 | 상태 |
|---|---|
| 핵심 스펙이 미결 결정 또는 확정된 선행 의존성 때문에 막힘 | 개입필요 |
| 오늘까지 결정해야 하는 안건이 실제 팀 대기를 발생 | 개입필요 |
| 핵심 마일스톤 초과 | 개입필요 |
| 하나의 블로커가 2명 이상 또는 2개 파트 이상을 막음 | 개입필요 |
| 확인된 실제 대기 관계 1건 이상 | 주의 |
| 범위 확정 예정일을 넘겼거나, 남은 기간이 확인된 후속 작업 소요기간보다 짧은데 범위 미확정 | 주의 |
| 선행 조건 충족 전 후속 작업 진행으로 재작업 위험 발생 | 주의 |
| 핵심 필드의 출처 충돌이 사람 확인 기한을 초과 | 주의 |
| 위 조건 없음 | 정상 |

`확인 필요`, `보고된 대기`, `판정 커버리지`는 건강 상태와 별도 축이다. 명시된 의존관계만 실제 병목에 포함하고, 파트 순서나 모호한 Slack 문장으로 추정한 관계는 `의존관계 확인 필요`로 표시하되 병목 수치에서 제외한다. 절대적인 `7일 무편집` 규칙은 v1에서 사용하지 않는다.

## GPT 요약 설계

### 역할 분리

`extract-slack.mjs`는 Slack 원문에서 구조화된 주장 후보만 만들고, `rules.mjs`와 `reconcile.mjs`가 확인 상태와 사실 데이터를 만든다. GPT 요약은 사실을 추가·삭제·재판정하지 않는다. GPT의 역할은 대표가 30초 안에 읽을 수 있도록 우선순위를 설명하고 문장을 압축하는 것이다.

| 담당 | 생성하는 내용 |
|---|---|
| 규칙·충돌 엔진 | 프로젝트 상태, 실제 대기, 재작업 위험, 영향 인원, 결정 기한, 출처 충돌, 전일 변화 |
| Slack 추출기 | 7개 유형의 주장 후보, 원문 발췌, 프로젝트·스펙 연결 후보, 명시성 등급 |
| GPT 요약 | 전체 헤드라인, 결정의 `왜 지금`, 권고안 요약, 병목·변화 설명, 선택된 Slack 신호 요약, 프로젝트 한 줄 요약 |
| UI | 사실 필드와 GPT 문장을 ID로 결합하고 Slack 원문·스레드 permalink와 확인 상태 표시 |

### AI 명령과 출력 계약의 위치

- Slack 주장 추출 지침: `prompts/slack-claim-extractor.md`
- Slack 추출 출력 계약: `schemas/slack-claim-extraction.schema.json`
- Slack 추출 실행·원문 부분 문자열·ID 검증: `lib/extract-slack.mjs`
- 브리핑 AI 행동 지침: `prompts/dashboard-summary.md`
- 브리핑 구조화 출력 계약: `schemas/dashboard-summary.schema.json`
- 브리핑 실행 및 검증: `lib/summarize.mjs`
- 입력 사실 생성: `lib/rules.mjs`, `lib/reconcile.mjs`, `lib/delta.mjs`

프롬프트를 바꾸면 문장과 우선순위 설명만 달라져야 한다. 상태·대기 인원·결정 기한처럼 운영 사실이 달라지면 안 된다.

`lib/extract-slack.mjs`가 Slack 원문을 제한된 구조로 추출하는 명령 지점이고, `lib/summarize.mjs`가 검증된 사실을 브리핑 문장으로 바꾸는 명령 지점이다. 두 단계는 서로 다른 프롬프트·스키마를 사용한다. 실행 흐름은 다음과 같다.

```js
const slackInstructions = readFile('prompts/slack-claim-extractor.md');
const slackSchema = readJson('schemas/slack-claim-extraction.schema.json');
const rawSlack = readJson('data/raw/slack.json');

const extracted = await extractionEngine.run({ instructions: slackInstructions, input: rawSlack, schema: slackSchema });
validateExcerptsAndCatalogIds(extracted, rawSlack);
writeJson('data/slack-claims.json', extracted);

const instructions = readFile('prompts/dashboard-summary.md');
const schema = readJson('schemas/dashboard-summary.schema.json');
const facts = readJson('data/dashboard-facts.json');

const narrative = await summaryEngine.run({ instructions, input: facts, schema });
validateIdsAndEvidence(narrative, facts);
writeJson('data/dashboard-narrative.json', narrative);
```

화면은 `dashboard-facts.json`의 사실과 `dashboard-narrative.json`의 문장을 ID로 합친다. 요약 엔진이 실패해도 facts 파일만으로 화면을 렌더링할 수 있어야 한다.

### GPT 입력

브리핑 GPT에는 Slack 원문 전체를 넘기지 않는다. 추출 단계에서 선택되고 원문 ID가 연결된 Slack 신호와 규칙 엔진이 확정한 최소 사실만 전달한다.

```json
{
  "asOf": "2026-07-14T07:30:00+09:00",
  "sourceFreshness": [],
  "sourceCoverage": [],
  "decisionReady": [],
  "needsContext": [],
  "bottlenecks": [],
  "deltas": [],
  "conflicts": [],
  "slackSignals": [],
  "projects": [],
  "peopleWaiting": [],
  "evidence": [{ "id": "ev-1", "source": "notion|slack|meeting", "url": "", "excerpt": "" }]
}
```

### GPT 출력

GPT는 `schemas/dashboard-summary.schema.json`에 맞춰 다음 서술 필드만 반환한다.

```json
{
  "headline": "오늘 바로 결정할 1건과 정보가 필요한 1건이 있습니다.",
  "decisionBriefs": [{ "decisionId": "dec-1", "whyNow": "", "recommendationSummary": "", "evidenceIds": ["ev-1"] }],
  "contextRequestBriefs": [{ "requestId": "ctx-1", "summary": "", "evidenceIds": ["ev-1"] }],
  "bottleneckBriefs": [{ "bottleneckId": "bn-1", "summary": "", "evidenceIds": ["ev-1"] }],
  "deltaBriefs": [{ "deltaId": "delta-1", "summary": "", "evidenceIds": ["ev-1"] }],
  "projectBriefs": [{ "projectId": "project-1", "oneLineSummary": "", "attentionReason": "", "evidenceIds": ["ev-1"] }],
  "slackSignalBriefs": [{ "signalId": "slack-signal-1", "summary": "", "evidenceIds": ["ev-1"] }],
  "dataWarnings": []
}
```

AI가 입력에 없는 ID·수치·담당자·기한을 만들거나, 서로 다른 출처 중 하나를 정답으로 고르면 출력 검증 실패로 처리한다. 모든 `evidenceIds`는 입력의 evidence에 존재해야 한다.

### 화면에서 보여줄 내용과 순서

상단 고정 신뢰 라인: 의존관계 검토율, Notion·Slack·회의록별 마지막 성공 시각, 채널 수집 성공률, 전일 스냅샷 존재 여부. 커버리지가 부족하면 `문제 없음`을 표시하지 않는다.

1. `오늘 결정할 것`: 대표의 선택이 후속 작업을 바꾸는 질문, 기한, 영향 스펙·작업, 원본 근거. 최대 5건
2. `결정 전 정보 필요`: 부족한 정보, 확인 대상, 확인 기한, 정보가 없을 때의 영향
3. `현재 막힌 것`: 실제 대기 체인, 대기 기간, 해소 행동, 담당자, 영향 범위
4. `어제와 달라진 것`: 새 병목, 위험 상승, 결정 완료, 병목 해소, 충돌 재개방, 핵심 스펙·기한 변경
5. `프로젝트·스펙`: 핵심 스펙 확정 여부, 파트 작업, 명시 의존관계, 후속 영향, 확인 상태와 Notion·회의록·Slack 근거
6. `담당별 흐름`: 자식 일감 기준 전원 업무량, 기한 초과·미설정, 특정 스펙 쏠림, 나를 기다리는 사람과 오늘 풀 수 있는 대기. 기본 정렬은 `내 작업을 기다리는 확인된 후속 작업 수`이며 관계 미측정 그룹 안에서만 열린 일감 수를 참고 정렬로 사용한다.
7. `확인 필요`: Notion·Slack·회의록의 서로 다른 주장과 담당 공백

개인 작업량은 병목 후보를 찾는 진입점으로 표시하되 성과·생산성 점수나 개인 순위를 만들지 않는다. 단순 완료율과 Slack 메시지 수도 대표 브리핑의 우선 요소로 사용하지 않는다.

### 실패와 폴백

- GPT 실패: 규칙 facts를 고정 템플릿으로 표시하고 `AI 요약 없음` 배지 노출
- GPT 출력 스키마 오류: 한 번 재시도 후 고정 템플릿 사용
- 근거 ID 오류 또는 새로운 수치 생성: 해당 서술 폐기
- Claude 폴백: 동일 프롬프트와 동일 스키마 사용, `engine=claude` 기록
- 출처 충돌: 합치지 않고 `확인 필요`로만 요약

## 스펙(기능) 단위 뷰

프로젝트 → 스펙(기능) → 파트 라인 3단 구조. 노션 작업 DB의 `상위 항목` 관계 = 스펙.

현재 운영 데이터는 2단 계층이다. `상위 항목`이 비어 있는 최상위 작업을 스펙으로 보고, `상위 항목`이 있는 작업을 해당 스펙의 자식 상세 일감으로 본다. 최상위 스펙에는 여러 직군 담당자가 함께 기록될 수 있으므로 그 담당자 목록을 개인 업무량으로 중복 집계하지 않는다.

```
피자레디
 └─ 익스프레스 기능            진척률 ▓▓▓░░ 40% · D-17 (7/31)
     기획  익스프레스 기획     담당 a · 7/11~7/31 · 진행 중
     개발  일부 선행 개발      담당 b · 기한 없음 ⚠ · 진행 중   ← ⚠ 기획 미완료 상태 개발 진행
     아트  머신 작업           담당 c · 진행 중
```

### 스펙 그룹핑
1. `상위 항목` 관계 있으면 그대로 사용 (정확)
2. 관계 없는 작업은 AI가 작업명·키워드로 기능 추정 그룹핑, `(추정)` 표시
3. 추정도 안 되면 "미분류" 그룹

### 스펙 카드 요소 (전부 채택)
| 요소 | 규칙 |
|---|---|
| 순서 이상 경고 | 같은 스펙에서 기획 미완료(또는 부재)인데 개발·아트 진행 중 → ⚠ 배지 |
| 진척률 바 | 하위 작업 완료 수 / 전체 수 |
| 기한 하이라이트 | D-3 이내 노랑, 초과 빨강, D-day 표시 |
| 정체 판단 | 절대 편집일이 아니라 계획 기준선 경과와 상태·단계 변화 여부로 판단. 기준선이 없으면 `판단 정보 부족` |
| 위생 경고 | 기한 미설정 · 담당자 미설정 회색 배지 |
| 통합 출처 근거 | 최근 Slack 원문 발췌·작성자·시각·스레드 링크·추출 주장·확인 상태를 해당 스펙에 연결. 메시지 수만 표시하지 않음 |

## 개인별 병목 뷰

프로젝트 목록의 `요약` 체크 대상에서 발견된 모든 담당자를 한 화면에 표시한다. 개인의 일감 수는 성과 평가가 아니라 업무 집중·기한 정리·팀 대기 가능성을 찾기 위한 운영 정보다.

- 계층 규칙: `상위 항목`이 없는 최상위 행은 스펙, `상위 항목`이 있는 행은 자식 상세 일감이다.
- 개인 업무량: 자식 상세 일감의 담당자 연결만 계산한다. 최상위 스펙의 담당자는 스펙 책임 맥락에는 사용하지만 개인 업무량에서는 제외한다.
- 공동 담당: 한 자식 일감에 여러 명이 있으면 각 개인에게 1건씩 연결하되 전체 자식 일감 수와 담당 연결 수를 구분한다.
- 표시 요소: 열린 자식 일감, 기한 초과, 기한 없음, 참여 프로젝트, 가장 집중된 스펙, 명시된 대기 관계.
- `내 작업을 기다리는 사람`: 후속 작업·사람, 대기 시작일, 해소 조건.
- `내가 기다리는 것`: 선행 작업·결정, 담당자, 결정 기한.
- `오늘 풀 수 있는 것`: 오늘 실행 가능한 확인·결정·전달 행동.
- `미지정 병목`: 담당자가 없는 자식 일감은 개인에게 귀속하지 않고 팀 큐로 표시한다.

기본 정렬은 `내 작업을 기다리는 확인된 후속 작업 수`다. 관계 데이터가 없는 사람은 `대기 영향 미측정`으로 묶고, 그 안에서만 열린 자식 일감 수를 참고 순서로 사용한다. 명시되지 않은 추정 의존관계는 실제 병목 수치에서 제외한다.

## 대시보드 화면 설계

모바일 우선 반응형 (1컬럼 → 데스크톱 그리드).

```
⓪ 수집 신뢰 상태       출처별 성공 시각 · 부분 실패 · 의존관계 커버리지 · 전일 스냅샷
① 오늘 결정할 것       질문 · 기한 · 영향 스펙/후속 작업 · 원본 근거 · 최대 5건
② 현재 막힌 것         실제 대기 · 재작업 위험 · 검수 대기 후보 · 판단 정보 부족
③ 어제와 달라진 것     스냅샷 기반 상태·기한·담당·완료율 변화. 스냅샷 없으면 비움
④ 프로젝트 탭          프로젝트 목록의 요약 체크 대상을 모두 표시
                       → 프로젝트 선택 칩(상태 점 + 스펙 수·자식 일감·미배정 수)
                       → 선택 시: 좌측 프로젝트 요약 카드(상태·진척·블로커·다음 액션·최근 회의)
                                 우측 스펙 리스트 (파트 라인 · 순서 이상 · D-day · Notion/회의록/Slack 근거)
                       모바일: 요약 카드 → 스펙 순 세로 배치
⑤ 담당별 흐름 탭       자식 일감 기준 전원 업무량 + 기한 초과·미설정 + 스펙 쏠림
                       + 누가 누구를 기다리는지 + 오늘 해소 행동 + 미지정 큐
⑥ 확인 필요 탭         출처별 주장 + 담당 공백 + 원본 링크
```

- 모든 화면은 읽기 전용이며 선택 저장·결정 기록·확인 요청 전송 기능이 없다.
- Slack은 별도 최상위 탭으로 분리하지 않고 해당 프로젝트·스펙의 근거와 확인 필요 항목에 붙인다.
- ①~③ = 첫 스크롤 안 (폰 기준). ④~⑥ = 탭 또는 하단 섹션
- 데이터 신선도 표시: 수집 시각 + 24h 초과 시 "오래된 데이터" 경고
- 엔진 폴백 발생 시: 상단 배너 "GPT 세션 만료 — Claude로 생성됨. codex login 후 scripts/upload-codex-auth.sh 실행"
