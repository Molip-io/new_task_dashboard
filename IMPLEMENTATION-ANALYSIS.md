# 업무 대시보드 개선 구현 전 분석

기준일: 2026-07-15  
기준 커밋: `b99d21a Establish a reproducible baseline for the decision dashboard`

## 1. Notion 작업관리 가이드에서 확인한 운영 규칙

업로드된 `노션_작업관리_가이드.pptx` 9장을 전체 확인했다.

- 계층은 상위 핵심 작업과 팀별 하위 작업의 2단계만 허용한다.
- 상위는 전체 시작일과 최종 완료일, 하위는 작업 기간과 데드라인을 기록한다.
- 완료 후 추가 작업은 기존 페이지를 다시 열지 않고 신규 페이지로 생성한다.
- 하위 작업 담당자는 현재 진행 현황과 액션 아이템을 수시로 갱신한다.
- 일정 지연 시 상위 페이지 댓글에 지연 사유, 이전 날짜와 변경 날짜, 관련 작업자와 PD·메인 기획자 태그, 문서 링크를 남긴다.
- Slack 공유는 가이드에 포함되지만 이번 구현에서는 발송·준수 검증을 하지 않는다.

## 2. 기존 코드 구조

### 화면

- `public/index.html`: 정적 헤더, KPI, 4개 탭 컨테이너만 제공한다.
- `public/app.js`: API 데이터 조회, 프로젝트·리스크·담당자·회의 화면 HTML을 문자열로 생성한다.
- `public/style.css`: 별도 디자인 시스템 없이 화면 전용 스타일을 제공한다.
- `decision-dashboard.html`: 서버 UI와 별개인 단일 파일 검토용 프로토타입이다.

컴포넌트 프레임워크나 라우터는 없다. 필터·정렬 상태 저장도 없다.

### 데이터 흐름

```text
Notion 프로젝트 목록 DB
  → 요약 체크 프로젝트 선택
Notion 작업 현황 DB 전체 검색
  → 프로젝트명 정규화
  → 일시 정지 계층 제외
  → 스펙/하위 작업 계층 구성
Slack 프로젝트 채널 + 최근 회의록 + 기존 요약 DB
  → dashboard.json
  → sourceHealth와 전일 snapshot diff 추가
  → public/app.js 표시
```

### 관계와 현재 집계

- 프로젝트는 프로젝트 목록 DB의 `요약` 체크로 선택한다.
- `parentIds`가 없으면 스펙, 있으면 하위 작업으로 본다.
- 담당자 통계는 완료되지 않은 하위 작업만 집계한다.
- 현재 담당자 정렬은 팀 대기 영향 수를 우선하며, 관계가 없으면 작업 수를 사용한다.
- 일시 정지 스펙과 그 하위 작업은 수집 결과에서 제외한다.

### 동기화와 캐시

- `data/dashboard.json`: 마지막 성공 수집 결과.
- `data/snapshots/YYYY-MM-DD.json`: 날짜별 비교 스냅샷.
- 데이터가 없으면 `data/dashboard.sample.json`을 반환한다.
- 서버 프로세스가 켜져 있을 때 `scheduleTime`과 로컬 시스템 시각이 같으면 하루 한 번 실행한다.
- 현재 스케줄러는 `Asia/Seoul`을 명시하지 않고 영업일도 구분하지 않는다.

### 현재 검증

- 미완료 작업의 마감일이 오늘 이전이면 기한 초과로 집계한다.
- Notion·Slack·회의록 수집 성공 여부와 의존관계 검토율을 표시한다.
- 전일과 상태·마감일·담당자·완료율 차이를 비교한다.
- 날짜 누락, 완료일 누락, 3단계 계층, 상하위 상태 불일치, 최신화 지연, 재작업 의심을 공통 이슈 모델로 만들지는 않는다.

## 3. Git 이력 분석

저장소에는 초기 기준선 커밋 `b99d21a`만 존재한다. `git log --follow`, `git show`, `git blame` 결과 수집기·계층·화면은 모두 이 커밋에서 처음 도입됐다.

- 관련 기능 최초 커밋: `b99d21a`
- 최근 수정 커밋: `b99d21a`
- 데이터 모델 또는 Notion 연동 변경 커밋: 별도 이력 없음
- 제거되거나 대체된 기능: 커밋 이력상 없음
- 임시 호환 코드: `notionSummary`와 `ai`의 선택적 폴백이 있으나 초기 기준선부터 존재한다.

따라서 이번 변경은 기존 의도를 보존하되 첫 구조 확장으로 기록한다. 팀 대기 영향도는 사용자 요구에 따라 UI와 계산을 함께 제거한다.

## 4. 발견한 문제와 영향 범위

1. `flatten()`의 people 값은 이름만 남겨 퇴사자 ID 기반 제외가 불가능하다.
2. 작업 데이터에 완료일·스프린트·페이지 생성일·프로젝트 연결 확실성·Git 활동이 없다.
3. 페이지 `last_edited_time`만 있어 상태 필드 변경일과 본문 진행 현황 변경일을 구분할 수 없다.
4. 작업 제목이나 관계 속성만 바뀐 경우도 Notion API의 페이지 수정 시각으로 잡힌다.
5. Git 저장소와 Notion 프로젝트/작업을 연결하는 설정이 없다.
6. 로컬 Git만으로 PR 정보는 얻을 수 없다. PR 링크는 향후 GitHub API 연동 영역이다.
7. 확인필요 화면이 프로젝트·문제 유형·스펙·작업 구조로 정규화되어 있지 않다.
8. 필터·정렬 상태가 화면 이동과 새로고침 후 유지되지 않는다.
9. 정기 수집이 KST 영업일 기준이 아니다.
10. 실제 Slack 알림 확장 이벤트가 없다.

## 5. 구현 경계와 설계 결정

- `ValidationIssue` 공통 구조를 도입하고 모든 화면이 동일 이슈 배열을 사용한다.
- 날짜 계산은 KST 날짜 문자열을 기준으로 하고 영업일 계산은 별도 모듈로 분리한다.
- Git은 설정된 로컬 저장소의 최근 커밋을 읽는다. 상태 자동 변경에는 사용하지 않는다.
- 프로젝트 연결은 저장소 설정의 프로젝트명으로, 작업항목 연결은 커밋 메시지의 `Git 키` 또는 Notion 작업항목 ID로 수행한다.
- 연결할 수 없는 Git 활동은 확인필요 이슈로 남긴다.
- 퇴사자 ID는 환경변수와 설정 배열에서 읽고, 담당자만 제거한다. 그 결과 담당자가 없어진 진행 중 작업은 재지정 이슈를 만든다.
- 상위·하위 기간 검증, 기한 초과, 완료일 누락, 계층 위반, 상태 불일치, 최신화 지연을 우선 구현한다.
- 완료 후 재오픈은 이전 스냅샷에 완료 상태가 있고 현재 미완료일 때 감지한다.
- Notion 댓글의 태그와 지연 사유는 현재 수집 데이터에 없으므로 구조화 필드가 존재할 때만 검증하고, 없다는 이유만으로 전 항목을 오류 처리하지 않는다.
- Slack 실제 발송은 만들지 않고 동기화 완료 결과 객체를 파일 이벤트로 남기는 확장 지점만 제공한다.

## 7. 검증 한계

- 실제 Notion 속성명과 사용자 ID는 인증된 재수집 전까지 확정할 수 없다.
- Notion API가 제공하는 페이지 수정 시각만으로 실질적인 현황 필드 수정 여부를 완전히 판별할 수 없다.
- 프로젝트별 Git 저장소 경로와 키워드가 설정되기 전에는 실제 게임 프로젝트 커밋을 비교할 수 없다.
- GitHub PR 정보는 별도 API 인증과 저장소 매핑이 필요하므로 이번 로컬 Git 수집 범위에는 포함하지 않는다.
