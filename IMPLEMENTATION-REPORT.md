# 업무 대시보드 개선 구현 보고

기준일: 2026-07-15  
구현 전 기준 커밋: `b99d21a Establish a reproducible baseline for the decision dashboard`

## 분석 결과

### 기존 구조

- `collect.mjs`가 Notion·Slack 수집, 프로젝트 카드, 담당자 집계를 한 파일에서 처리했다.
- `public/app.js`가 프레임워크 없이 프로젝트·리스크·담당자·회의 화면을 렌더링했다.
- Notion people 속성은 이름만 남겨 사용자 ID 기반 퇴사자 제외가 불가능했다.
- 검증은 기한 초과와 출처 상태, 전일 상태·기한·담당 차이에 한정됐다.
- Git 활동 수집, 필수 속성 검사, 공통 검증 이슈, 필터 상태 보존은 없었다.

### 관련 Git 커밋과 변경 이력

- `git log`, `git log --follow`, `git show`, `git blame`로 관련 파일 이력을 확인했다.
- 관련 기능은 모두 루트 커밋 `b99d21a`에서 처음 도입됐다.
- 이후 데이터 모델·Notion 연동 변경, 기능 제거, 호환성 코드 도입 이력은 없었다.
- 기존 읽기 전용·출처 보존 원칙을 유지하면서 첫 도메인 모델 확장으로 구현했다.

### 발견한 문제

- 완료일·스프린트·Notion 사용자 ID·Git 키가 수집 모델에 없었다.
- 2단계 계층, 날짜, 완료일, 최신화, 재오픈, 상하위 상태를 검증하지 않았다.
- Git 커밋과 Notion 작업항목의 불일치를 확인할 방법이 없었다.
- 프로젝트 미연결 작업과 Git 활동이 조용히 누락될 수 있었다.
- 담당자 화면에 제거 요청된 팀 대기 영향 계산과 표현이 남아 있었다.
- 스케줄러가 `Asia/Seoul`을 명시하지 않았다.

### 변경 영향 범위

- 수집: Notion people 원본 ID, 작업 속성, Git 저장소, Slack 대화, 회의록.
- 규칙: 공통 `ValidationIssue`, KST 날짜·영업일, 스냅샷 비교, 프로젝트·담당자 집계.
- 화면: 4개 메뉴, KPI, 프로젝트 계층, 담당자 상세, 프로젝트별 확인필요, 공통 필터·정렬.
- 운영: 오전 1회 KST 수집, 동기화 완료 이벤트 파일, 설정·문서·샘플.

## 구현 내용

### 변경한 기능

- 메뉴를 `브리핑 / 프로젝트 / 담당자 / 확인필요`로 변경했다.
- 사용자 화면의 작업 단위를 `작업항목`으로 통일했다.
- 브리핑에 프로젝트·작업항목·기한·날짜·완료일·최신화·가이드·Git 지표 10개와 출처 신뢰선을 추가했다.
- 프로젝트에서 목표, 완료율, 관리 상태, 스펙 토글, 작업항목 상세를 표시한다.
- 담당자에서 프로젝트 수, 전체·진행 중 작업항목, 기한 초과, 최신화 필요, 가까운 마감일을 표시하고 상세 작업을 토글한다.
- 확인필요를 `프로젝트 → 문제 유형 → 스펙·작업항목`으로 그룹화하고 미연결 항목은 `프로젝트 미분류`로 올린다.
- 정렬·필터 상태는 localStorage와 URL에 보존한다.

### 추가한 검증 규칙

- 상위·하위 2단계 초과, 존재하지 않는 상위, 프로젝트·스펙 연결 누락.
- 시작일·마감일·완료일·담당자 누락, 시작일/마감일 역전, 상하위 기간 불일치.
- 미완료 마감일 초과와 달력 일수 기준 초과 일수.
- 상하위 완료 상태 불일치와 전일 완료 후 재오픈.
- 진행 중 작업의 KST 영업일 기준 최신화 지연.
- 지연 사유, 변경 전후 일정, PD·메인 기획자 공유 태그 누락.
- Notion 갱신 후 Git 활동, 완료일 이후 Git 활동, 작업항목·프로젝트에 연결되지 않은 커밋.
- 필수 Notion DB 속성 세팅 상태.

### 제거한 기능

- `팀 대기 영향도`의 화면, 정렬, 집계, 계산, ViewModel 응답 필드를 제거했다.
- 기존 독립 실행형 프로토타입은 서버 대시보드 안내 페이지로 축소해 오래된 계산·용어가 노출되지 않게 했다.

### 정렬 및 필터 처리 방식

- 현재 데이터 규모에 맞춰 브라우저의 단일 ViewModel 모듈에서 처리한다.
- 프로젝트 기본 정렬은 확인 이슈, 기한 초과, 최신화, 가까운 마감일, 이름 순이다.
- 담당자 기본 정렬은 기한 초과, 최신화, 진행 중 작업항목, 이름 순이다.
- 작업항목 기본 정렬은 위험도, 상태, 마감일, 이름 순이다.
- 상태·팀·프로젝트·스펙·스프린트·담당자·기간·기한 초과·최신화·가이드·확인 유형·완료 포함·Git 활동·Git 불일치를 조합할 수 있다.

### Git과 Notion 연결 방식

- `config.json`에 프로젝트별 로컬 Git 저장소 경로와 Notion 프로젝트명을 설정한다.
- 최근 커밋 일시, 작성자, 메시지, 변경 파일, 현재 브랜치, 원격 커밋 URL을 읽는다.
- 커밋 메시지에 Notion `Git 키` 또는 작업항목 ID가 있으면 작업항목에 연결한다.
- 연결이 불확실하면 상태를 변경하지 않고 `UNMAPPED_GIT_ACTIVITY`로 표시한다.

- 게임 프로젝트 로컬 저장소 경로가 제공되지 않아 실제 프로젝트 커밋과 Notion 데이터 비교는 실행하지 못했다.
- 실제 Slack 메시지 발송은 요구사항대로 구현·시험하지 않았다.

## 남은 확인사항

- Notion API의 `last_edited_time`은 페이지 전체 수정 시각이므로 작업현황 필드·상태 필드·제목/관계 수정만을 완전히 구분하지 못한다.
- Notion 댓글 본문과 댓글 속 태그는 현재 수집 모델에 포함되지 않는다. 지연 사유·이전 일정·공유 담당자를 구조화 속성으로 두면 검증할 수 있다.
- 로컬 Git만으로 GitHub PR 정보를 안정적으로 얻을 수 없다. PR 링크는 GitHub API 인증과 저장소 설정이 추가로 필요하다.
- 커밋 메시지에 작업 키가 없으면 프로젝트까지만 연결되고 작업항목은 확인필요로 남는다.
- 실제 프로젝트 저장소를 `config.json`에 연결하고 1회 인증 수집 후 Notion 실제 속성 별칭과 데이터량을 확인해야 한다.
- 추후 Slack notifier는 `dashboardSyncCompleted` 이벤트를 구독하고 `리더의_공간`, 한국 영업일, Webhook/토큰 환경변수를 설정해야 한다.
